# 17. 오류 분석 + 점수 정체 진단 프롬프트
목적: 중반(9~12h) 점수 정체 시 "다음에 뭘 해야 점수가 오르는가"를 데이터로 결정.

===== 프롬프트 시작 (A. 오류 분석) =====

너는 Kaggle Grandmaster다. 현재 베스트 모델의 OOF를 해부해 점수 향상 방향을 도출하는 오류 분석 코드를 작성하라.

[입력] oof 확률, folds.csv, 04번 피처, {평가지표}

[요구사항]
1. confusion matrix (세그먼트/녹음 단위 각각) + 정규화 버전,
   가장 헷갈리는 클래스 쌍 top-3와 해당 오분류 파일 목록 저장 (청취 검수용 — 보조 팀원)
2. 클래스별 F1 분해: 어느 클래스가 전체 macro-F1을 깎는지 + 그 클래스의
   precision/recall 중 무엇이 문제인지 -> 대응 방향 자동 출력
   (recall 낮음 -> 증강/오버샘플/threshold, precision 낮음 -> 혼동 클래스와의 구분 피처)
3. 오분류 샘플 프로파일링: 맞춘 샘플 대비 오분류 샘플의
   duration/SNR(spectral flatness)/RMS/DEMON 피크 강도 분포 비교
   -> "짧은 파일에서 약함", "저SNR에서 약함" 같은 패턴을 수치로 보고
4. 세그먼트 vs 녹음 단위 불일치 분석: 세그먼트 다수결과 다른 녹음 목록
   -> 집계 방식(mean/max/trimmed) 재튜닝 근거
5. 모델 간 불일치 분석: 보유한 OOF들(lgbm/cnn/ast)이 서로 다르게 예측한 샘플 군집
   -> 앙상블 기대 효과 추정 (oracle 앙상블 상한 점수 계산)
6. 캘리브레이션: reliability diagram + ECE, temperature scaling 적용 시 점수 변화

===== 프롬프트 끝 (A) =====

===== 프롬프트 시작 (B. 점수 정체 진단 트리) =====

너는 대회 전략 코치다. 아래 상태 정보를 받아 진단 트리에 따라 "지금 가장 기대값 높은 다음 액션 1~3개"를 우선순위와 예상 소요 시간과 함께 제시하라. 동의만 하지 말고 근거를 요구하라.

[상태 입력 양식]
- 남은 시간: {h}
- CV(녹음 단위) / public 점수 이력: {표}
- CV-public 갭 추이: {확대/유지/축소}
- 보유 모델과 OOF 상관행렬: {값}
- 클래스별 F1: {값}
- 시도했으나 실패한 것: {목록}

[진단 트리 — 이 로직으로 판단하라]
1. CV-public 갭 확대 중 -> CV 신뢰 붕괴. 분할/누수/분포이동 재점검(03, 16번)이 최우선.
   새 모델 추가는 중단.
2. CV는 오르는데 public 정체 -> 분포 이동 의심. 16번 레벨1부터.
3. 둘 다 정체 + 모델 간 OOF 상관 > 0.95 -> 다양성 고갈. 새 표현(LOFAR/CQT 브랜치,
   14번 임베딩 추가)이 튜닝보다 기대값 높음.
4. 둘 다 정체 + 특정 클래스 F1만 낮음 -> 해당 클래스 표적 대응:
   오버샘플/합성(15번), 이진 전문가 모델(해당 클래스 vs rest) 추가 후 확률 보정.
5. 둘 다 정체 + 전 클래스 고른 부진 -> 입력 표현 자체 한계.
   세그먼트 길이/sr/f_max 재검토 (짧으면 LOFAR 해상도 부족 — 긴 세그먼트 실험).
6. 시간 < 6h -> 신규 아키텍처 금지. 앙상블 최적화/집계 튜닝/seed 앙상블만.
